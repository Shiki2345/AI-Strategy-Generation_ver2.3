# 平台能力地图产出说明

> 产出日期：2026-04-14
> 涉及文件：platform_module_map.json（新建）、skill_definition.json（更新）、提示词.md（更新）

---

## 一、背景与问题

### 改动前的问题

当用户提出的需求超出访问策略的能力范围时，Agent 的响应存在两个问题：

**问题 1：展示技术方案而非行动指引**

用户说"我想让每个人只能看到自己创建的订单"，Agent 回复：

> 这个需求超出了平台访问策略的能力范围。
> **替代方案：** 在业务系统后端或数据库层实现行级数据过滤（Row-Level Security），让查询自动注入 `WHERE created_by = 当前用户` 的条件。这需要开发团队在订单管理理模块中实现，不通过紫鸟平台处理。

对小白用户来说，Row-Level Security、WHERE created_by 等于天书。用户看完的反应是"这是什么？我该怎么办？"，而不是"我知道下一步去哪了"。

**问题 2：Agent 只了解自己的模块，无法指路**

Agent 只知道"访问策略设置"这一个模块的能力边界。当用户的需求属于紫鸟平台其他模块（如屏幕水印、登录控制、事中监管等）时，Agent 只能说"这个不归我管"，无法告诉用户具体去哪里操作。

相当于一个只知道自己工位的前台——被问到任何其他问题，只能说"找别人"。

### 目标

让 Agent 拥有一张紫鸟平台的"地图"，能够：
1. 用大白话解释为什么做不了
2. 精确指路到平台内的具体模块（给出导航路径）
3. 超出整个平台能力时，告诉用户去找谁

### 置信度分级

收集过程中区分了两级置信度：

| 级别 | 含义 | 判定标准 | 数量 |
|------|------|---------|------|
| **verified** | 已通过页面截图确认功能 | 看到了实际页面内容、功能文案、或完整的子菜单结构 | 13 个模块 |
| **inferred** | 仅看到菜单项，功能为合理推断 | 只看到菜单名称，未进入实际页面 | 11 个模块 |

标注为 inferred 的模块，handles 描述仅保留从菜单名可合理推断的内容，不编造未确认的功能细节。例如"屏幕水印设置"只写 `"配置屏幕水印（具体功能待确认）"`，不编造"显示员工姓名、时间"等未经确认的细节。

## 三、产出物：platform_module_map.json

### 文件位置

`access_policy_skill_package/platform_module_map.json`

### 数据结构

每个模块包含以下字段：

```json
{
  "module_id": "唯一标识",
  "module_name": "模块名称（与平台菜单一致）",
  "nav_path": "导航路径（如：安全监管 → 安全拦截 → 访问策略设置）",
  "confidence": "verified | inferred",
  "confidence_source": "verified 时注明信息来源",
  "is_current_agent_scope": true,
  "handles": ["该模块能做的事"],
  "route_when_user_says": ["用户说了什么时应该指向这个模块"]
}
```

### 统计

| 类别 | 数量 |
|------|------|
| 总模块数 | 24 个 |
| 安全监管类 | 9 个 |
| 企业管理类 | 7 个 |
| 账号/设备类 | 2 个 |
| 云号 | 1 个 |
| "更多"菜单类 | 4 个 |
| 费用管理 | 1 个 |
| 指路规则数 | 17 条 |

### routing_rules 设计

routing_rules 是地图的核心——定义了"用户想做 X → 应该去哪里"的映射关系。

每条规则包含：

| 字段 | 说明 |
|------|------|
| `user_intent` | 用户想做什么 |
| `route_to` | 指向的模块 ID 或外部系统 |
| `nav_path` | 平台内的导航路径（平台内模块才有） |
| `who_to_contact` | 超出平台能力时该找谁（外部系统才有） |
| `first_check` | 需要先判断条件的场景 |
| `platform_alternative` | 平台能部分满足时的替代建议 |
| `also_suggest` | 额外建议 |

## 四、关联改动

### skill_definition.json 改动

**改动 1：新增地图引用**

```json
"platform_module_map_ref": "platform_module_map.json — 紫鸟平台完整能力地图，当用户需求超出访问策略能力时，根据该地图的 routing_rules 指引用户到正确的模块"
```

**改动 2：not_supported 数据结构重构**

经历两轮迭代：

第一轮（v2.1）将单一 `alternative` 拆为 `user_explanation` + `where_to_go` + `who_to_contact`。
第二轮（v2.2）去掉了 `where_to_go` 和 `who_to_contact`（不再指向外部系统和外部角色），改为 `platform_suggestion`（平台内替代方案）+ `if_cannot_solve`（坦诚告知做不到）。

最终结构：
```json
{
  "capability": "业务系统字段级权限",
  "description": "控制某个字段是否可见/可编辑（如：隐藏价格列）",
  "user_explanation": "访问策略可以控制整个页面的访问权限，但没办法只隐藏页面里的某几个字段。",
  "platform_suggestion": "如果您想隐藏的是页面上的某个按钮或特定区域，可以试试「网页元素采集」功能...",
  "if_cannot_solve": "如果需要隐藏的是表格里的某一列数据（比如价格、地址），这个目前做不到。"
}
```

**改动 3：on_not_supported 响应模板重写**

改动前：
```
"response_template": "这个需求超出了平台访问策略的能力范围。{{description}}。建议的替代方案：{{alternative}}"
```

改动后：
```
"response_template": "{{user_explanation}}\n\n💡 **平台内的替代方案：**{{platform_suggestion}}\n\n💡 **我能帮你做的：**{{platform_can_do_instead}}"
```

`response_tone` 约束更新为：
- 建议只推荐紫鸟平台内已有的模块和功能
- 不推荐外部系统（ERP、OA等）
- 不出现「找技术团队」「找服务商」等指向外部角色的引导
- 确实做不到就坦诚告知「目前做不到」

**改动 4：conditionally_supported（共用账号场景）**

改动前：
```json
"alternative": "在业务系统后端/数据库层实现 Row-Level Security"
```

改动后：
```json
"user_explanation": "因为你们所有人共用一个卖家账号，大家打开同一个页面看到的内容是一样的...",
"platform_suggestion": "如果可以给不同组分配独立的平台账号，我就能帮您通过访问策略实现组间的页面隔离。您可以到「账号管理 → 全部账号」中查看是否能拆分账号。",
"if_cannot_split": "在共用账号的情况下，这个需求目前做不到。如果后续拆分了独立账号，随时来找我配置。"
```

### 提示词.md 改动

"不支持的能力"部分经历两轮迭代：

第一轮（v2.1）从 4 条扩展到 11 条，每条带导航路径，但仍包含外部系统引导。
第二轮（v2.2）确立核心原则「Agent 的建议边界 = 紫鸟平台的能力边界」，去掉所有外部系统和外部角色引导。

最终版本（11 条，全部限定在平台内）：
- 数据按人隔离 → 建议拆分独立账号（「账号管理 → 全部账号」），无法拆分则告知做不到
- 隐藏字段 → 先判断「网页元素采集」能否实现，不行则告知做不到
- 多级审批 → 告知平台支持两种方式，主动问是否够用
- API 权限 → 引导到「企业管理 → 企业设置 → 开发者模式配置」
- 防截图/防拍照 → 引导到「安全监管 → 屏幕水印设置」
- 监控员工操作 → 引导到「安全监管 → 安全监管 → 事中监管设置」
- 查看拦截记录 → 引导到「安全监管 → 安全拦截 → 访问拦截日志」
- 查看操作/登录记录 → 引导到「更多 → 企业日志」
- 限制登录/设备控制 → 引导到「更多 → 登录控制」
- 接收验证码/号码管理 → 引导到「紫鸟云号」

## 五、效果对比

### 场景 1：用户要求"订单页面隐藏收货地址和电话"

改动前：
> 这个需求超出了平台访问策略的能力范围。控制某个字段是否可见/可编辑（如：隐藏价格列）。建议的替代方案：在业务系统的角色权限模块中配置。

改动后：
> 我理解你的需求：让运营能正常打开订单页面，但看不到客户的收货地址和电话号码。
>
> 紫鸟平台有一个「网页元素采集」功能，可以对页面内的特定区域进行隐藏或限制查看。如果地址和电话是独立的页面区域，可以通过这个功能实现。请问你们用的是哪个平台的订单页面？
>
> 如果元素采集无法精准选中，说明这些信息在页面上和其他内容混在一起，这个目前做不到。

### 场景 2：用户要求"防止员工截图泄露客户信息"

改动前：
> 我是访问策略配置助手，只能帮你配置和管理访问策略。如果你有策略需求，请描述你想限制的页面、人员和时段。

改动后：
> 防截图泄露不在访问策略的范围内，但紫鸟平台有专门的功能可以处理。
>
> 您可以到「安全监管 → 屏幕水印设置」中配置屏幕水印。

### 场景 3：用户要求"共用账号下只看自己创建的订单"

改动前：
> 替代方案：在业务系统后端/数据库层实现 Row-Level Security（行级数据过滤），让查询自动注入 WHERE created_by = 当前用户 的条件。

改动后：
> 因为你们所有人共用一个卖家账号，大家打开同一个页面看到的内容是一样的。紫鸟平台可以控制「谁能打开哪个页面」，但没办法控制「同一个页面里谁能看到哪些数据行」。
>
> 如果可以给不同组分配独立的平台账号，我就能帮您通过访问策略实现组间的页面隔离。您可以到「账号管理 → 全部账号」中查看是否能拆分账号。
>
> 如果没办法拆分账号，这个需求目前做不到。后续拆分了独立账号，随时来找我配置。
