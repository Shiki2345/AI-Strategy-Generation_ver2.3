### 第1步：用户输入 → 意图理解

**用户说**：
```
"不允许客服身份的成员来访问美国亚马逊的账号管理相关页面"
```

**LLM处理（Prompt Engineering）**：
```python
prompt = f"""

请以JSON格式返回：
{{
    "platform": "平台名称",

    "roles": ["受限角色"],
```

## 🧠 知识库 vs RAG：关系详解

### 什么是知识库？

**知识库**就是一个**结构化的信息仓库**，像一个超级详细的字典：

```
知识库 = 数据文件 + 数据结构

例如：
{
  "平台": "亚马逊",
  "功能模块": "账号管理",
  "相关URL": [
    "https://amazon.com/...",
    "https://amazon.com/..."
  ],
  "风险等级": "高",
  "推荐限制": ["客服", "实习生"]
}
```

**特点**：
- ✅ 信息结构化、准确
- ✅ 可以直接读取
- ❌ 但AI不能"理解"自然语言去查询

---

### 什么是RAG？

**RAG = Retrieval-Augmented Generation**  
（检索增强生成）

**本质**：让AI能够"查字典"

传统LLM的问题：
```
用户: "亚马逊账号管理的URL是什么？"
GPT: "抱歉，我不知道具体URL..." ❌
      （因为训练数据中没有这个信息）
```

使用RAG后：
```
用户: "亚马逊账号管理的URL是什么？"

RAG系统：
  1. 理解问题："亚马逊 + 账号管理"
  2. 查询知识库（通过向量相似度搜索）
  3. 找到相关信息
  4. 把信息交给LLM
  5. LLM基于检索到的信息生成回答

AI: "根据知识库，亚马逊账号管理包含以下URL：
     1. https://amazon.com/gp/css/homepage.html
     2. ..."  ✅
```

---

### RAG的核心技术：向量检索

#### 为什么需要向量？

**问题**：用户的表达千变万化

```
用户可能说：
- "账号管理"
- "账户设置"  
- "修改账号信息"
- "个人中心"
- "Account Settings"

知识库里只存了："账号管理"

如何匹配？
```

**解决方案**：向量化

```python
# 将文字转换为向量（数字表示）
embedding_model.embed("账号管理")
# → [0.23, -0.45, 0.78, ..., 0.12]  (1536维向量)

embedding_model.embed("账户设置")
# → [0.25, -0.43, 0.76, ..., 0.11]  (非常接近！)

embedding_model.embed("商品管理")
# → [-0.12, 0.67, -0.34, ..., 0.89]  (差距很大)

# 计算相似度
similarity("账号管理", "账户设置") = 0.95  ✅ 高度相似
similarity("账号管理", "商品管理") = 0.45  ❌ 不相似
```

## 🔧 技术栈选型

### 向量数据库选择

| 数据库 | 优点 | 缺点 | 适用场景 |
|--------|------|------|----------|
| **Chroma** | • 轻量级<br>• 易部署<br>• 开源免费 | • 性能一般<br>• 不支持分布式 | 小型项目<br>本地开发 |
| **Pinecone** | • 性能强<br>• 云端服务<br>• 自动扩展 | • 收费<br>• 需要联网 | 生产环境<br>大规模应用 |
| **Milvus** | • 开源<br>• 高性能<br>• 支持分布式 | • 部署复杂<br>• 资源消耗大 | 企业级应用 |

**推荐**：
- 快速验证：**Chroma**（今天就能跑起来）
- 生产环境：**Pinecone**（稳定可靠）

### Embedding模型选择

| 模型 | 优点 | 缺点 |
|------|------|------|
| **OpenAI Embedding** | 效果好、简单 | 收费、需联网 |
| **HuggingFace模型** | 免费、本地运行 | 中文效果一般 |
| **BGE系列（中文）** | 免费、中文效果好 | 需要本地部署 |

**推荐**：OpenAI Embedding（简单高效）

```python
def simple_match(user_input):
    # 关键词匹配
```

### 阶段2：引入RAG（3-5天）

**目标**：支持模糊匹配、同义词

```python
from langchain.vectorstores import Chroma

# 初始化

# 加载知识库
knowledge = load_json("amazon_knowledge.json")
for module in knowledge['modules']:
    text = f"{module['name']} {module['description']}"
    vectordb.add_texts([text], metadatas=[module])

# 使用
def rag_match(user_input):
    results = vectordb.similarity_search(user_input, k=3)
    urls = []
    for doc in results:
        urls.extend(doc.metadata['urls'])
    return urls

# 测试
urls = rag_match("客服不能看账户设置")  # 注意：用户说的是"账户设置"
print(urls)  # ✅ 仍然能匹配到"账号管理"的URL
```

## ❓ 常见问题

### Q1: 知识库需要多大？
**A**: 取决于平台数量
- 1个平台（如亚马逊）：约100-200个URL
- 5个主流平台：约500-1000个URL
- 足够覆盖80%的常见需求

### Q2: 向量数据库会很大吗？
**A**: 不会
- 1000个URL的向量数据：约10-20MB
- 笔记本电脑就能运行

### Q3: 用户说的话知识库里没有怎么办？

```python
if similarity_score < 0.7:  # 相似度太低

         A. 账号管理
         B. 支付管理
         C. 订单管理

```

**A**: 三种方式

2. **自动学习**：从用户手动添加的URL中学习
3. **爬虫采集**：定期爬取平台页面结构变化

## 🎯 总结

### 核心流程（5步）

```
用户说话 → 意图理解 → RAG检索知识库 → 多轮对话 → 生成JSON
   ↓           ↓            ↓              ↓          ↓
  LLM      Prompt      向量数据库    对话管理   格式化输出
```

### 知识库 + RAG 的关系

```
知识库 = 存什么（内容）
RAG = 怎么找（方法）

就像：
知识库 = 图书馆的所有书
RAG = 图书馆的检索系统
```

### 关键技术点

1. **Prompt Engineering**：让LLM理解用户意图
2. **向量化 (Embedding)**：让机器理解语义相似性
3. **相似度搜索**：从知识库中找到最匹配的信息
4. **多轮对话**：补充缺失信息
5. **Skill沉淀**：让系统越用越聪明
